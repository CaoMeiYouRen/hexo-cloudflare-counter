# hexo-cloudflare-counter 设计文档

## 1. 项目背景

LeanCloud 相关服务逐步停服后，原有基于 LeanCloud 的 Hexo 阅读量方案难以继续使用。大量站点仍依赖 hexo-leancloud-counter、hexo-leancloud-counter-security 及其生态中的模板代码，因此本项目的核心目标不是重新定义一套全新协议，而是以最小迁移成本提供一个可持续维护的后端与前端方案。

本项目计划使用 Cloudflare D1 作为首选后端存储，同时优先补齐本地 SQLite 实现，以支持不同部署环境。在运行时层面，项目第一阶段聚焦 Cloudflare Workers 与 Docker/普通 Node.js 自托管；Vercel 由于生产场景不适合直接依赖本地 SQLite，暂不作为当前阶段交付目标，后续如需支持，应结合 PostgreSQL、MySQL 等外部数据库重新设计。项目还可额外提供一个自有 Hexo 插件，降低对陈旧前端插件的依赖。

## 2. 设计目标

### 2.1 核心目标

1. 完全兼容 leancloud_visitors 常用 API，保证已有主题模板和插件在最少改动下继续工作。
2. 尽可能兼容 hexo-leancloud-counter-security 的调用方式，至少为后续兼容预留明确扩展点。
3. 抽象数据库访问层，支持 D1、SQLite，并为后续扩展其他数据库保留能力。
4. 抽象运行时适配层，优先支持 Workers 与 Docker/普通 Node.js 自托管，并为后续接入 Vercel + 外部数据库保留扩展能力。
5. 提供自有 Hexo 插件作为推荐前端接入方式，同时保留 LeanCloud 兼容模式。

### 2.2 非目标

1. 不实现完整 LeanCloud 平台能力，仅兼容阅读量场景所需的最小 REST API 子集。
2. 不在第一阶段支持复杂 ACL、用户体系、云函数、通用数据存储等 LeanCloud 功能。
3. 不为了兼容性而牺牲可维护性；兼容层应与核心计数服务解耦。

## 3. 用户与场景

### 3.1 目标用户

1. 现有 Hexo 博客站长，已使用 hexo-leancloud-counter 或其衍生插件。
2. 希望将计数服务从第三方 SaaS 迁移到自托管环境的用户。
3. 需要在 Cloudflare Workers、Docker 或普通 Node.js 环境中运行统一服务的开发者。

### 3.2 典型使用场景

1. 老站点仅替换 API 地址和应用密钥，即继续使用原有主题模板统计阅读量。
2. 新站点直接使用本项目提供的 Hexo 插件，以更现代的配置方式完成文章阅读量展示与累加。
3. 开发环境使用本地 SQLite，生产环境切换到 D1，代码层无需改业务逻辑。

## 4. 总体设计原则

1. 兼容优先：先兼容现有调用，再提供原生能力。
2. 分层清晰：运行时、路由、业务、存储、兼容层相互隔离。
3. 低迁移成本：尽量复用既有前端模板、请求格式和字段命名。
4. 多运行时一致性：不同部署平台对外行为一致，差异收敛在适配层。
5. 可测试性：核心业务逻辑与运行时、数据库驱动解耦，便于单元测试与集成测试。

## 5. 功能范围

### 5.1 MVP 范围

1. 提供 LeanCloud Counter 最小兼容接口。
2. 支持单篇文章查询、新建、计数自增。
3. 支持文章列表批量查询阅读量。
4. 支持 D1 和 SQLite。
5. 支持 Workers、Docker 与普通 Node.js 自托管。
6. 提供旧 LeanCloud Counter 数据迁移脚本。
7. 提供基础鉴权、限流、日志和健康检查。

### 5.2 第二阶段范围

1. 兼容 hexo-leancloud-counter-security 的安全调用模型。
2. 提供自有 Hexo 插件。
3. 提供管理接口，如重建索引、数据修复、迁移导入导出。
4. 支持更多数据库实现，如 PostgreSQL、MySQL、Turso。
5. 在外部数据库方案明确后再恢复 Vercel 支持。

## 6. API 兼容设计

### 6.1 兼容目标

兼容对象以现有主题和插件中最常见的 LeanCloud REST 调用为主。从实际使用方式看，阅读量统计主要依赖以下三类请求：

1. 按 URL 查询计数。
2. 新建一条 Counter 记录。
3. 按 objectId 对 time 字段执行增量更新。

因此兼容层应实现 LeanCloud API 的“阅读量最小子集”，而非完整复刻。

### 6.2 推荐兼容接口

兼容层建议统一挂载在 /1.1 下，并至少支持以下端点：

1. GET /1.1/classes/Counter?where=...
2. POST /1.1/classes/Counter
3. PUT /1.1/classes/Counter/:objectId

建议保留以下请求与响应约定：

1. 请求头兼容 X-LC-Id、X-LC-Key、Content-Type。
2. 查询参数支持 where，至少支持 { url: string } 与 { url: { "$in": string[] } }。
3. 新建记录支持字段 title、url、time。
4. 更新记录支持 { time: { "__op": "Increment", "amount": number } }。
5. 查询结果返回 { results: CounterRecord[] }。
6. 新建和更新返回 objectId、createdAt、updatedAt 等 LeanCloud 风格字段。

### 6.3 数据模型兼容

兼容字段建议如下：

| 字段 | 含义 | 说明 |
| --- | --- | --- |
| objectId | 记录唯一 ID | 对外暴露的兼容主键，可与内部主键一致，也可单独映射 |
| url | 文章唯一地址 | 兼容老插件核心查询字段 |
| title | 文章标题 | 兼容已有前端创建逻辑 |
| time | 阅读量 | 自增计数字段 |
| createdAt | 创建时间 | LeanCloud 风格返回字段 |
| updatedAt | 更新时间 | LeanCloud 风格返回字段 |

### 6.4 hexo-leancloud-counter-security 兼容策略

该项应作为增强兼容能力处理，而非阻塞 MVP。设计上预留 SecurityAdapter，负责解析和校验安全版插件附加的签名、时间戳或中间层规则。文档阶段先明确以下原则：

1. 核心计数服务不直接耦合某个安全插件的签名实现。
2. 安全逻辑以中间件或适配器形式接入。
3. 若安全版插件与标准 LeanCloud API 存在差异，应在兼容层进行请求转换。
4. 若部分历史行为难以完全复刻，应优先提供“配置兼容模式 + 文档迁移指引”。

#### 6.4.1 插件现有行为分析

从现有仓库实现看，hexo-leancloud-counter-security 不是单纯修改前端请求，而是把“Counter 记录初始化”前置到 Hexo 构建和部署流程中。它主要包含以下几部分：

1. Generator 阶段遍历已发布文章，生成 leancloud_counter_security_urls.json，内容包含 title 和 url 列表。
2. Deployer 阶段读取该 JSON 文件，并通过 leancloud-storage SDK 登录 LeanCloud。
3. 同步时按 url 查询 Counter 记录，不存在则创建，初始 time 为 0。
4. 插件使用 leancloud.memo 记录已成功同步的文章，避免重复请求并降低 429 风险。
5. 插件额外提供 hexo lc-counter register 和简写命令，用于注册后台同步所需用户。

换言之，该插件修复的核心思路不是改变浏览器侧统计 API，而是禁止前端在记录不存在时直接创建 Counter，改为在部署期由受信任账号预先创建数据。

#### 6.4.2 与 NexT 安全模式的关系

在 NexT 主题的安全模式下，前端模板通常只负责读取和递增已存在的 Counter 记录；当记录不存在时，不再由浏览器直接发起创建请求，而是提示 Counter 尚未初始化。因此，hexo-leancloud-counter-security 实际上依赖两部分同时成立：

1. 主题侧开启 leancloud_visitors.security 配置，关闭前端自动创建记录。
2. 部署侧通过插件同步所有文章 URL，保证 Counter 在上线前已存在。

这意味着本项目若要兼容该生态，不能只讨论 HTTP API，还必须覆盖“初始化数据从哪里来”这个流程问题。

#### 6.4.3 兼容层级划分

该插件的兼容应拆成三个层级，避免范围失控：

1. 主题兼容：兼容 security 模式下的前端访问行为，确保记录不存在时不会要求浏览器侧创建。这一层级应纳入第二阶段范围。
2. 工作流兼容：提供等价的文章预同步能力，例如管理接口、CLI 或 Hexo 扩展，在部署期批量创建缺失的 Counter。这一层级是推荐实现路径。
3. SDK 级完全兼容：兼容该插件当前基于 leancloud-storage 的登录、注册、查询、保存流程。这意味着后端需要额外模拟 LeanCloud 用户体系和 SDK 所依赖的接口子集，复杂度明显高于普通 REST 兼容，不建议纳入 MVP。

#### 6.4.4 对本项目的实现启示

基于上述分析，本项目更合理的方案如下：

1. 保持对 leancloud_visitors 安全模式的前端兼容。
2. 提供一个原生的后台初始化能力，用于接收文章列表并批量创建 Counter。
3. 将 leancloud.memo 的语义提升为“已同步文章索引”，但具体存储形式不必与历史插件完全一致，可以改为数据库表或服务端状态文件。
4. 在自有 Hexo 插件中内建同步逻辑，替代对 leancloud-storage SDK 和 LeanCloud 用户系统的依赖。
5. 对旧项目提供迁移文档，说明如何从 hexo-leancloud-counter-security 迁移到本项目的原生同步流程。

#### 6.4.5 建议新增的管理能力

若项目要覆盖安全插件的实际需求，建议在公开计数 API 之外增加一组管理能力，仅供部署期或管理员调用：

1. 批量导入文章标题和 URL。
2. 批量创建缺失 Counter。
3. 查询某批 URL 的初始化状态。
4. 可选地导出或重建同步索引。

这组能力应与公开读取和计数 API 分离，避免安全边界混杂。

## 7. 系统架构

### 7.1 分层结构

建议采用如下分层：

1. Transport 层：负责 HTTP 协议处理、跨域、超时、限流、错误格式。
2. Compatibility 层：负责 LeanCloud API 兼容与安全插件兼容。
3. Service 层：负责计数查询、新建、递增、幂等约束等核心业务。
4. Repository 层：负责统一数据库读写接口。
5. Runtime Adapter 层：负责 Workers、Node、Docker 等环境差异，并为后续接入 Vercel + 外部数据库保留适配点。

### 7.2 推荐目录演进

从当前阶段到后续插件落地，目录结构建议分两步演进：

1. MVP 阶段允许继续保持单包结构，以便优先完成后端 API、SQLite/D1 和迁移脚本。
2. 在开始实现并发布自有 Hexo 插件之前，建议切换为 workspace 结构，避免“同一个 npm 包既像后端又像插件”的语义混乱；发布基础设施可以更早按 workspace 目标形态准备。

推荐的 workspace 形态如下：

1. apps/server：后端服务入口，负责 Node.js、Docker、Workers 等部署物。
2. packages/core：共享业务内核，包含兼容层、Service、Repository、迁移逻辑与类型定义。
3. packages/hexo-plugin：自有 Hexo 插件，负责主题注入、前端请求封装和部署期同步命令。
4. packages/runtime-workers：可选，封装 Workers/D1 相关入口与适配代码。
5. packages/runtime-node：可选，封装 Node.js/SQLite 相关入口与适配代码。
6. scripts：仓库级脚本，如发布、迁移样例处理、基准测试等。

若在插件开发前暂未完成完整包拆分，则至少应在代码组织上预留 core 层，并提前准备 pnpm workspace、发布脚本和 CI 工作流，避免后续拆分时发生大规模迁移。

### 7.3 交付形态与包策略

项目的核心定位应明确为“后端阅读量服务”，而不是“Hexo 插件集合”。在此基础上再提供插件和部署适配，用户心智会更稳定。

#### 7.3.1 主交付形态

1. 项目主交付物是后端 API 服务，负责提供 LeanCloud 兼容接口、管理接口和迁移能力。
2. Workers、Docker、普通 Node.js 用户消费的都是服务端部署物，而不是 Hexo 插件包。
3. 自有 Hexo 插件属于配套客户端能力，用于改善新用户接入体验，但不改变项目本体是后端服务这一定位。

#### 7.3.2 npm 包发布策略

1. 服务端不应以“Hexo 插件”身份发布到 npm，否则会误导用户把部署型项目理解成博客端依赖。
2. 自有 Hexo 插件应作为独立 npm 包发布，供 Hexo 站点通过 npm 安装和配置。
3. 服务端如需发布到 npm，应以库或 CLI/服务包的身份发布，其文案、README 和 package metadata 都应明确这是后端服务。
4. Docker 镜像、Workers 部署配置和 Node.js 运行入口都应围绕服务端主包构建，而不是围绕插件包构建。

#### 7.3.3 维护策略

1. 后端服务与 Hexo 插件应共享同一套业务协议，但不共享发布身份。
2. 共享逻辑应下沉到 core 包，避免插件直接依赖服务端入口实现。
3. 插件版本可以跟随后端协议演进独立发布，不必与服务端发布节奏完全绑定。
4. 旧用户迁移文档应优先说明“如何部署后端服务”；插件安装说明应作为新接入方式单独提供。

#### 7.3.4 是否立即引入 workspace

1. MVP 阶段若仍以单包业务实现为主，可以暂不立即完成全部包拆分，优先完成后端可用性。
2. 但发布配置、CI 和目录约束可以提前按 workspace 目标形态准备，例如引入 pnpm workspace 和独立的包发布工作流。
3. 一旦开始实现并准备发布自有 Hexo 插件，就应完成 workspace 或等价 monorepo 结构拆分。
4. 这样可以在不打断当前开发节奏的前提下，避免后续出现一个包同时承担“服务端部署物”和“Hexo 插件”的混淆。

#### 7.3.5 发布与版本策略

项目进入 workspace 形态后，服务端与插件不应继续共用同一种发布工具和同一条版本线。

1. 根级后端服务继续使用 semantic-release，负责服务端版本号、Git 标签、GitHub Release 以及与服务端交付物相关的自动化流程。
2. workspace 中对外发布的 npm 包使用 Changesets，负责多包场景下的版本计算、changelog 生成与 npm 发布。
3. 内部共享包如 packages/core 可保持 private，不强制进入公开发布流程。
4. CI 工作流需要拆分为“服务端发布”和“包发布”两条流水线，分别处理不同交付物。
5. 服务端发布不应再承担 workspace 包的版本编排；包发布也不应反向驱动 Docker、Workers 等服务端部署语义。

### 7.4 核心调用流程

以单篇文章访问计数为例：

1. 前端插件或主题模板向兼容接口发起 GET /classes/Counter?where={url}。
2. 兼容层解析 where 并转为统一查询对象。
3. Service 层根据 url 查询记录。
4. 若记录存在，返回 results[0]，前端随后发起自增请求。
5. 若记录不存在，前端发起 POST 创建记录。
6. PUT 请求通过 objectId 定位记录，并执行 time 自增。

对于首页批量展示场景：

1. 前端向 GET /classes/Counter?where={ url: { "$in": [...] } } 发起批量查询。
2. 兼容层解析数组条件并调用批量查询接口。
3. Service 层返回匹配记录集合。
4. 前端自行将缺失记录显示为 0。

## 8. 存储设计

### 8.1 数据库抽象接口

Repository 层建议至少提供以下抽象能力：

1. findByUrl(url)
2. findByUrls(urls)
3. createCounter(input)
4. incrementCounterByObjectId(objectId, amount)
5. findByObjectId(objectId)
6. upsertCounterByUrl(input)

其中前四项为 MVP 必需能力。

### 8.2 表结构建议

建议定义单表 counters：

| 字段 | 类型 | 说明 |
| --- | --- | --- |
| id | integer / text | 内部主键 |
| object_id | text | 对外兼容 objectId，唯一索引 |
| url | text | 文章 URL，唯一索引 |
| title | text | 文章标题 |
| time | integer | 阅读量，默认 0 |
| created_at | datetime | 创建时间 |
| updated_at | datetime | 更新时间 |

建议约束：

1. url 唯一，避免重复创建。
2. object_id 唯一，保证兼容更新接口。
3. time 不允许为负值。
4. 对 url 建立查询索引。

### 8.3 D1 与 SQLite 适配

两类数据库都属于 SQL 模型，业务上应共享同一套 Repository 接口，不同点仅体现在驱动初始化和 SQL 方言细节：

1. D1 用于 Workers 生产部署，优先保证无状态、边缘部署友好。
2. SQLite 用于本地开发、Docker 单机部署和测试。
3. SQL 语句应尽量使用两者共通语法。
4. 事务、返回值格式和参数绑定差异由 Driver 层封装。

### 8.4 并发与一致性

阅读量自增是核心写入路径，需保证并发下结果正确：

1. 使用数据库原子更新语句执行 time = time + amount。
2. 创建记录时依赖 url 唯一约束，避免并发写入重复记录。
3. 当新建发生唯一键冲突时，可回退到查询已有记录。
4. 业务层不依赖内存锁，以适配多实例部署。

### 8.5 旧数据迁移脚本设计

为了降低旧站迁移成本，MVP 应提供一个面向 LeanCloud Counter 导出数据的迁移脚本，用于将历史阅读量直接导入本项目当前使用的数据库。

#### 8.5.1 输入格式约定

迁移脚本首期只需要支持 LeanCloud 常见导出格式中的 Counter JSONL 文件，格式特征如下：

1. 文件第一行可能是以 #filetype:JSON-streaming 开头的元信息头，用于描述类名和导出类型。
2. 后续每一行是一个独立 JSON 对象，对应一条 Counter 记录。
3. 迁移脚本至少需要识别 objectId、title、url、time、createdAt、updatedAt。
4. ACL、xid 等与阅读量核心逻辑无关的字段可忽略，但不应导致解析失败。
5. 仅处理 class=Counter 的导出内容，其他类数据不在当前范围内。

#### 8.5.2 导入字段映射

建议采用如下字段映射：

| 导出字段 | 目标字段 | 说明 |
| --- | --- | --- |
| objectId | object_id | 保留原 LeanCloud 主键，便于兼容后续 PUT 更新 |
| url | url | 作为文章唯一标识和主要去重依据 |
| title | title | 原样迁移，缺失时允许为空字符串 |
| time | time | 原样迁移，缺失时按 0 处理 |
| createdAt | created_at | 尽量保留原始创建时间 |
| updatedAt | updated_at | 尽量保留原始更新时间 |

#### 8.5.3 导入模式与覆盖语义

结合当前阶段假设，迁移脚本默认面向“目标平台尚无历史数据”的首次导入场景，但仍需要明确覆盖语义并向用户发出风险提示：

1. MVP 迁移脚本采用全量导入思路，默认导入前清空 counters 表中的现有数据。
2. 清空并覆盖属于破坏性操作，脚本在执行前应显式提醒用户核对目标数据库路径、环境和备份状态。
3. 若用户误将脚本指向已有生产数据，现有数据会被覆盖，因此文档必须明确该风险。
4. 后续若需要保留增量合并能力，可再补充 upsert 模式，但不作为当前阶段阻塞项。

#### 8.5.4 去重与幂等要求

即使在“全量覆盖导入”模式下，脚本仍需要具备稳定的去重和幂等特性：

1. 同一份导出文件重复执行两次，最终数据库结果应保持一致，不能产生额外重复记录。
2. 同一导出文件内部若出现重复记录，应优先按 url 去重；若 url 缺失，再回退到 objectId 判断。
3. 对重复项可采用“保留最后一条”或“保留 updatedAt 较新一条”的策略，但实现时应固定规则，避免结果不稳定。
4. 导入完成后，url 唯一约束和 object_id 唯一约束应继续成立。

#### 8.5.5 脚本职责边界

迁移脚本的首期职责应尽量收敛，避免演变为通用 ETL 工具：

1. 负责读取 JSONL、解析 Counter 记录、完成去重和导入。
2. 负责输出导入总数、跳过数、重复数、失败数等摘要信息。
3. 不负责迁移 LeanCloud ACL、用户体系、Class Schema 等非 Counter 数据。
4. 不负责在线双写、断点续传或跨多数据库类型自动同步。

## 9. 运行时与部署设计

### 9.1 当前阶段运行时目标

项目应通过统一业务内核支持多运行时：

1. Cloudflare Workers：首要部署目标，默认搭配 D1。
2. Docker：面向传统自托管，默认可搭配 SQLite。
3. 普通 Node.js：面向本地开发、单机部署或其他可持久化文件系统环境，默认可搭配 SQLite。
4. Vercel：暂不纳入当前阶段交付范围，待外部数据库方案明确后再支持。

### 9.2 运行时适配原则

1. 路由与业务逻辑保持一致，不因平台不同而出现 API 差异。
2. 平台差异仅体现在启动入口、环境变量读取、数据库绑定方式。
3. 日志、健康检查、错误格式保持一致。
4. 平台专属能力由入口文件注入，而不是分叉业务代码。
5. 对无法稳定提供持久化本地文件系统的平台，不应强行复用 SQLite 方案。

### 9.3 推荐部署组合

| 部署方式 | 推荐数据库 | 说明 |
| --- | --- | --- |
| Workers | D1 | 官方主路径，边缘部署成本低 |
| Docker | SQLite | 自托管最简方案，便于迁移与备份 |
| Node.js 自托管 | SQLite | 适合本地开发、单机服务和迁移落地 |
| Vercel | PostgreSQL / MySQL 等外部数据库 | 当前阶段暂缓，待外部数据库接入后再恢复支持 |

### 9.4 配置项建议

建议统一使用环境变量描述运行时能力：

1. DB_DRIVER：d1、sqlite。
2. D1 绑定名或连接配置。
3. SQLITE_PATH：SQLite 文件路径。
4. APP_ID / APP_KEY：LeanCloud 兼容模式下的应用标识。
5. SECURITY_MODE：是否启用安全兼容模式。
6. MIGRATION_SOURCE：迁移脚本输入文件路径。
7. MIGRATION_RESET：是否允许执行清空后全量导入。
8. CORS_ALLOW_ORIGIN、RATE_LIMIT、LOG_LEVEL 等通用配置。

## 10. 自有 Hexo 插件设计

### 10.1 设计动机

继续兼容老插件可以降低迁移成本，但长期看，原有插件年久失修、前端逻辑分散、主题耦合较强。本项目应提供一个自有 Hexo 插件，作为更推荐的接入方案。

### 10.2 插件目标

1. 以更现代的配置方式接入本项目后端。
2. 支持文章页自动自增、列表页批量查询。
3. 兼容主流 Hexo 主题的模板注入方式。
4. 提供原生模式和 LeanCloud 兼容模式两种接入方式。

### 10.3 插件职责边界

插件建议只负责前端展示与请求封装，不承担业务安全和数据校验。其职责包括：

1. 生成页面所需 DOM 标记与脚本。
2. 统一请求后端 API。
3. 处理列表页和详情页的阅读量展示。
4. 在必要时兼容旧主题中的 leancloud_visitors CSS class 或 data attribute。

### 10.4 插件接入模式

1. LeanCloud 兼容模式：复用旧模板结构，请求本项目提供的兼容 API。
2. 原生模式：直接调用项目自定义 API，减少历史包袱，便于后续迭代。

### 10.5 从安全插件迁移

针对 hexo-leancloud-counter-security 用户，自有 Hexo 插件应提供明确迁移路径：

1. 保留文章页和列表页的阅读量展示行为。
2. 提供部署期同步命令，替代旧插件依赖的 leancloud-storage 登录与同步流程。
3. 支持读取 Hexo 已发布文章列表并批量初始化 Counter。
4. 在文档中说明旧版 leancloud.memo 的作用，并给出迁移后是否继续保留本地同步索引的策略。

### 10.6 插件发布与仓库关系

自有 Hexo 插件虽然属于本项目生态的一部分，但在交付上不应与后端服务混为同一个发布物。

1. 插件应作为独立 npm 包发布，并在说明中明确其依赖本项目后端服务或兼容 API。
2. 插件仓库可与后端服务共用一个 monorepo/workspace，以复用类型、协议和测试设施。
3. 插件不应直接复用服务端启动入口或运行时脚本，避免依赖边界混乱。
4. 若未来提供原生模式与 LeanCloud 兼容模式两种接入方式，应优先通过插件配置切换，而不是通过发布不同插件包解决。
5. 插件版本与服务端版本可以分别演进，二者通过兼容协议而不是共享版本号保持协作。

## 11. 安全与运维设计

### 11.1 基础安全

1. 兼容接口可通过 APP_ID / APP_KEY 做基础鉴权。
2. 对写接口启用限流，降低刷量风险。
3. 校验 Referer、Origin 或签名作为可选增强手段。
4. 对异常输入、超长 body、非法 where 进行严格校验。

### 11.2 防刷量策略

阅读量天然难以做到绝对准确，项目应在“兼容现有行为”和“控制恶意刷量”之间取得平衡。建议提供可配置策略：

1. 基于 IP + URL 的短时去重。
2. 基于 User-Agent 的简单过滤。
3. 为安全兼容模式预留签名校验扩展点。
4. 管理端支持查看异常请求统计。

### 11.3 可观测性

1. 记录请求日志、错误日志、慢查询日志。
2. 提供健康检查端点。
3. 为不同运行时统一 requestId，便于排障。
4. 将数据库错误转换为稳定的 API 响应格式。
5. 对迁移脚本输出结构化日志，便于用户核对覆盖范围和导入结果。
6. 对服务端发布和包发布分别保留独立的 CI 日志与制品记录，便于排查发布故障。

### 12.1 单元测试

1. 兼容层 where 解析与请求映射。

2. 批量查询流程。
3. SQLite 驱动集成测试。
4. 迁移脚本对 SQLite 的全量覆盖导入测试。
5. 不同运行时入口的最小冒烟测试。

### 12.3 兼容性测试

1. 使用现有 leancloud_visitors 模板脚本进行端到端验证。
2. 针对 hexo-leancloud-counter-security 补充兼容样例。
3. 构造旧主题模板页面，验证无需修改或少量修改即可运行。

### 阶段一：MVP 后端

1. 完成 Counter 数据模型与 SQLite 驱动。
2. 完成 LeanCloud 最小兼容接口。
3. 完成旧 LeanCloud Counter JSONL 迁移脚本。
4. 完成基础测试与 Docker 启动方案。

### 阶段二：Workers 与 D1

1. 完成 D1 驱动接入。
2. 完成 Workers 部署配置。

### 阶段三：外部数据库与安全兼容

1. 评估并接入 PostgreSQL、MySQL 等外部数据库。

3. 增加限流、签名、去重等安全能力。
4. 在外部数据库方案稳定后再恢复 Vercel 部署适配。

### 阶段四：自有 Hexo 插件

1. 完善 workspace 或等价 monorepo 结构，拆分 server、core、hexo-plugin 等包。

3. 兼容旧主题 DOM 结构。
4. 将插件作为独立 npm 包发布，并补充迁移文档与示例站点。

## 14. 风险与决策点

1. hexo-leancloud-counter-security 的历史实现细节需要进一步调研，否则只能做到部分兼容。
2. 若追求对 hexo-leancloud-counter-security 的原插件零修改兼容，则不仅要兼容 REST API，还要兼容 leancloud-storage SDK 所使用的用户登录、注册、查询和保存能力，实施成本较高。
3. Vercel 对本地 SQLite 的生产可行性有限，因此当前阶段应明确暂缓 Vercel，而不是给出误导性的“可直接支持”预期。
4. 迁移脚本若采用全量覆盖模式，用户误操作会清空目标数据，因此需要明显的执行前确认和备份提示。
5. LeanCloud 查询语法非常宽泛，若追求完全兼容会显著增加复杂度，因此应明确仅支持阅读量场景所需子集。
6. 若未来提供原生 API，需要避免与兼容 API 混杂，建议从一开始区分 /1.1 与 /api 两套入口。
7. 若继续把服务端和插件维持在同一个发布包中，后续 README、版本、依赖与用户接入方式都会持续混淆，因此应尽早在设计上拆分发布身份。
8. 若在 workspace 中继续用单一 semantic-release 流程统管所有包，后续 tag、版本范围和包间依赖处理会显著复杂化，因此更适合拆分为 semantic-release + Changesets 双轨策略。

## 15. 结论

本项目的正确方向不是简单替换数据库，而是构建一个“兼容旧生态、同时具备长期可维护性”的阅读量服务。技术上应采用“LeanCloud 兼容层 + 统一计数服务 + 多数据库驱动 + 多运行时适配 + 自有 Hexo 插件”的架构。这样既能解决旧站迁移问题，也能为新站提供更可控的接入方式。

从实施顺序看，建议先完成 LeanCloud 最小兼容接口、SQLite/D1 双存储以及旧 Counter 数据迁移脚本，再扩展安全兼容和自有 Hexo 插件。项目本体应始终保持“后端服务优先”的定位；发布基础设施可以提前按 workspace 形态演进，其中根级服务端继续使用 semantic-release，对外 npm 包改用 Changesets。Vercel 支持应放到外部数据库能力明确之后处理。这样可以用最短路径恢复旧站点可用性，同时保持后续演进空间。
